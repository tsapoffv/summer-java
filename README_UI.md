# README_UI.md — Руководство по пользовательскому интерфейсу клиента «Морской бой»

## 1. Введение

Пользовательский интерфейс (UI) клиента реализован на **Java Swing** — стандартной библиотеке для построения графических приложений, входящей в состав Java SE. Выбор Swing обусловлен:

- **Кроссплатформенностью** — приложение одинаково работает на Windows, macOS и Linux.
- **Отсутствием внешних зависимотелей** — не требуется дополнительных библиотек, что упрощает сборку и запуск.
- **Гибкостью** — Swing позволяет легко создавать кастомные компоненты с переопределённой отрисовкой.
- **Интеграцией с AWT** — доступ к Java 2D для высококачественной графики.

Интерфейс построен как **однооконное приложение** (`JFrame`), внутри которого с помощью `CardLayout` переключаются четыре логических экрана (панели):

1. **Подключение** (`ConnectionPanel`)
2. **Расстановка кораблей** (`SetupPanel`)
3. **Лобби** (`LobbyPanel`)
4. **Бой** (`BattlePanel`)

Каждая панель реализует свой набор функций и взаимодействует с сетевым слоем через методы главного класса `BattleshipClientGUI`.

## 2. Главное окно — `BattleshipClientGUI`

**Файл:** `client/BattleshipClientGUI.java`

Этот класс является точкой входа клиента. Он наследует `JFrame` и содержит:

- `CardLayout` и контейнер `mainPanel` для хранения четырёх панелей.
- Ссылки на сокет и потоки ввода/вывода.
- Поток для приёма сообщений (`receiverThread`).
- Флаг `connected` (volatile) для управления состоянием.
- Методы: `connect()`, `disconnect()`, `send()`, `showPanel()`, `handleMessage()`.

### 2.1. Управление экранами: `CardLayout`

`CardLayout` — это менеджер компоновки, который хранит несколько компонентов в стеке и показывает только один за раз. Он идеально подходит для приложений-мастеров или последовательных экранов, как в нашем случае.

**Почему выбран `CardLayout`?**

- **Простота** — достаточно вызвать `cardLayout.show(panel, name)`.
- **Сохранение состояния** — панели не уничтожаются, их состояние (поля, данные) сохраняется при переключении.
- **Быстрое переключение** — без задержек на создание/уничтожение компонентов.
- **Альтернативы** — `JTabbedPane` не подходит (не нужны вкладки), ручное управление видимостью (setVisible) менее удобно.

**Использование:**
```java
cardLayout = new CardLayout();
mainPanel = new JPanel(cardLayout);
mainPanel.add(connectionPanel, "CONNECT");
mainPanel.add(setupPanel, "SETUP");
mainPanel.add(lobbyPanel, "LOBBY");
mainPanel.add(battlePanel, "BATTLE");
// ...
public void showPanel(String name) {
    cardLayout.show(mainPanel, name);
    mainPanel.revalidate();
    mainPanel.repaint();
}
```

### 2.2. Сетевое взаимодействие

Клиент использует **TCP-сокеты** (`java.net.Socket`). Для обмена данными применяются `DataInputStream` и `DataOutputStream` — они обеспечивают атомарное чтение/запись строк в кодировке UTF-8.

- **Подключение** запускается в отдельном потоке (`Connector`) из `ConnectionPanel`, чтобы не блокировать EDT.
- После успешного подключения стартует **поток-приёмник** (`receiverThread`) с флагом `daemon = true`. Он непрерывно читает входящие команды и передаёт их в EDT через `SwingUtilities.invokeLater()`.

**Почему `invokeLater`?** Swing — однопоточный, все изменения GUI должны выполняться в **Event Dispatch Thread (EDT)**. Без `invokeLater` может возникнуть `ConcurrentModificationException`, мерцание или некорректное отображение.

**Метод `handleMessage()`** — центральный диспетчер. Он анализирует команды от сервера и вызывает соответствующие методы панелей. Все вызовы происходят в EDT, поэтому можно безопасно изменять компоненты.

### 2.3. Потокобезопасность

- `connected` объявлен `volatile` для гарантии видимости между потоками.
- Все обращения к GUI — только через `invokeLater`.
- Отправка команд (`send()`) выполняется в любом потоке (обычно EDT), но сам `writeUTF()` может блокироваться, однако это не критично, т.к. сетевой ввод-вывод обычно быстрый.

## 3. Панель подключения — `ConnectionPanel`

**Файл:** `client/panels/ConnectionPanel.java`

### 3.1. Назначение

Предоставляет форму для ввода адреса и порта сервера, отображает статус подключения, запускает процесс соединения.

### 3.2. Компоновка: `GridBagLayout`

`GridBagLayout` выбран для точного позиционирования элементов с разной шириной колонок, отступами и центрированием.

**Структура:**
- Заголовок (2 колонки)
- Подзаголовок (2 колонки)
- Метка «Сервер:» (1 колонка) + поле ввода (1 колонка)
- Метка «Порт:» + поле ввода
- Кнопка «Подключиться» (2 колонки, с увеличенным отступом сверху)
- Индикатор статуса (кружок + текст)

**Почему не другие менеджеры?**
- `GridLayout` — все ячейки одинаковой ширины, что некрасиво для формы с метками и полями.
- `BoxLayout` — только одномерный, сложно сделать центрирование по вертикали.
- `SpringLayout` — слишком сложен и громоздок.
- `GridBagLayout` даёт полный контроль над позиционированием и отступами.

### 3.3. Компоненты

- **Поля ввода** — `RoundedTextField` (кастомный компонент) со скруглёнными углами и рамкой, меняющей цвет при фокусе.
- **Кнопка** — `GradientButton` с градиентом и эффектами наведения/нажатия.
- **Статус** — `JLabel` с текстом и `JLabel`-индикатор (кружок «●»), цвет меняется в зависимости от состояния.

### 3.4. Взаимодействие

При нажатии кнопки:
1. Кнопка блокируется.
2. Статус меняется на «Подключение...».
3. Запускается поток `Connector`, который вызывает `client.connect()`.
4. При успехе — статус «Ожидание запроса поля...», при ошибке — разблокировка кнопки и сообщение об ошибке.

## 4. Панель расстановки — `SetupPanel`

**Файл:** `client/panels/SetupPanel.java`

### 4.1. Назначение

Позволяет игроку расставить 10 кораблей на поле 10×10 вручную или автоматически, а затем отправить готовое поле на сервер.

### 4.2. Компоновка: `BorderLayout`

Структура:
- **NORTH**: заголовок и информационная строка (`lblInfo`).
- **CENTER**: `BorderLayout` с двумя областями:
  - **CENTER**: игровое поле (`BoardPanel` в режиме `SETUP`) в обёртке с заголовком.
  - **EAST**: палитра кораблей (`shipPalette`) в обёртке с заголовком (ширина 200px).
- **SOUTH**: панель с четырьмя кнопками управления (FlowLayout).

`BorderLayout` выбран, потому что он естественно делит окно на основные зоны: заголовок сверху, главный контент в центре, кнопки снизу. Внутри центра используется ещё один `BorderLayout` для размещения поля и палитры слева и справа.

### 4.3. Палитра кораблей

`shipPalette` — это `JPanel` с `GridLayout(0,1,8,8)`, т.е. вертикальный список с отступами. Каждый элемент — кастомный `JPanel` (`shipItem`), содержащий:

- Визуализацию корабля (вложенный `JPanel` с переопределённым `paintComponent` для рисования закрашенных прямоугольников).
- Счётчик оставшихся кораблей данного размера (`×N`).

**Почему `GridLayout` для палитры?**
- Элементы должны располагаться строго один под другим.
- `BoxLayout` также подошёл бы, но `GridLayout` проще и даёт одинаковую высоту ячеек.

При клике на элемент палитры:
- Устанавливается `selectedShipSize`.
- Обновляется информационная строка.

### 4.4. Игровое поле — `BoardPanel`

Этот компонент подробно описан в разделе 6. В режиме `SETUP` он:
- Отображает расставленные корабли.
- Показывает подсветку при наведении мыши (зелёную, если размещение возможно, и красную, если нет).
- Обрабатывает клики для размещения корабля.

### 4.5. Кнопки управления

- **«Повернуть (R)»** — меняет флаг `horizontal` и перерисовывает поле.
- **«Авто»** — генерирует случайное поле (рекурсивный алгоритм) и заполняет им `field`.
- **«Очистить»** — сбрасывает расстановку.
- **«Готов»** — отправляет поле на сервер (активируется только когда все корабли расставлены).

### 4.6. Обработка мыши

- **`mouseClicked`** на поле — проверяет `selectedShipSize`, координаты, вызывает `canPlace()` и при успехе размещает корабль.
- **`mouseMoved`** — вызывает `board.setHighlight()` для предпросмотра.
- **Горячая клавиша R** — реализована через `InputMap`/`ActionMap`, что позволяет реагировать на нажатие независимо от фокуса.

### 4.7. Автоматическая расстановка

Алгоритм:
1. Список размеров перемешивается.
2. Для каждого размера делается до 5000 попыток случайного размещения (случайные координаты и ориентация).
3. При неудаче — рекурсивный перезапуск всей процедуры.
Это гарантирует нахождение валидного поля, т.к. пространство решений конечно.

## 5. Панель лобби — `LobbyPanel`

**Файл:** `client/panels/LobbyPanel.java`

### 5.1. Назначение

Показывает список доступных игроков, позволяет обновить список и вызвать выбранного игрока на бой.

### 5.2. Компоновка: `BorderLayout`

- **NORTH**: заголовок «Лобби» + подзаголовок.
- **CENTER**: `JScrollPane` со списком игроков (`JList<PlayerItem>`).
- **SOUTH**: статусная строка и панель с двумя кнопками.

### 5.3. Список игроков: `JList` + `DefaultListModel`

`JList` — стандартный компонент для отображения списка. В сочетании с `DefaultListModel` он обеспечивает динамическое обновление: при изменении модели список автоматически перерисовывается.

**Почему `JList`, а не `JTable`?**
- Нужен простой список, без колонок.
- `JList` легче и проще в настройке.

**Модель данных:** `PlayerItem` — простой POJO с полями `id` и `name`.

### 5.4. Кастомный рендерер — `PlayerCellRenderer`

Расширяет `JPanel` и реализует `ListCellRenderer<PlayerItem>`. Это позволяет полностью контролировать отображение каждого элемента:
- Иконка «👤» слева.
- Имя и ID — в вертикальном `GridLayout` (2 строки).
- При выборе элемента рисуется полупрозрачный золотой фон и рамка.

**Почему кастомный рендерер?**
- Стандартный рендерер не позволяет легко разместить иконку и текст в две строки с разными шрифтами.
- Кастомный рендерер даёт полный контроль над внешним видом, включая состояние выделения.

### 5.5. Кнопки

- **«Обновить»** — отправляет `GET_PLAYERS`, обновляет список.
- **«Вызвать на бой»** — отправляет `SELECT_PLAYER:<id>` выбранному игроку.

## 6. Панель боя — `BattlePanel`

**Файл:** `client/panels/BattlePanel.java`

### 6.1. Назначение

Главный игровой экран: отображение двух полей, ходы, лог событий, кнопка сдачи.

### 6.2. Компоновка: `BorderLayout`

- **NORTH**: баннер статуса (с иконкой, текстом и счётчиками кораблей). Баннер — кастомный `JPanel` с переопределённой отрисовкой для скруглённых углов и рамки.
- **CENTER**: `GridLayout(1,2,20,0)` с двумя обёртками полей.
- **SOUTH**: `BorderLayout` с `EventLog` в центре и кнопкой «Сдаться» внизу.

`GridLayout` для полей даёт им равную ширину (50/50), что необходимо для симметричного отображения.

### 6.3. Статусный баннер

Содержит:
- Иконку (`lblStatusIcon`) — меняется в зависимости от состояния (⚔, 🎯, ⏳).
- Текст статуса (`lblStatus`) — крупный жирный шрифт, цвет меняется (зелёный — ваш ход, серый — ожидание, жёлтый — дополнительный ход).
- Счётчик кораблей — `lblShipsRemaining` с информацией о количестве непотопленных кораблей у игрока и противника.

### 6.4. Игровые поля — два `BoardPanel`

Один в режиме `MY_FIELD` (отображает свои корабли и попадания/промахи противника), другой в режиме `ENEMY_FIELD` (корабли противника скрыты, видны только свои выстрелы). Подробно о `BoardPanel` — в разделе 6.

### 6.5. Обработка выстрелов

- На поле противника навешиваются `MouseListener` и `MouseMotionListener`.
- При клике проверяется `inGame && myTurn`, координаты и что клетка не обстреляна.
- Отправляется `MOVE:x:y`.
- Результат применяется через `applyMyMove()` (для игрока) и `applyOpponentMove()` (для противника), которые обновляют массивы `enemyCells`/`myCells` и перерисовывают соответствующие поля.

### 6.6. Лог событий — `EventLog`

Отдельный компонент (описан ниже), который выводит сообщения с цветовой кодировкой.

### 6.7. Кнопка «Сдаться»

Отправляет `SURRENDER`, сбрасывает флаги `inGame` и `myTurn`.

## 7. Вспомогательные компоненты

### 7.1. `BoardPanel` — кастомная отрисовка игрового поля

**Файл:** `client/components/BoardPanel.java`

Это самый сложный визуальный компонент. Он полностью переопределяет `paintComponent()` для рендеринга клеток с использованием **Java 2D**.

#### 7.1.1. Режимы (`Mode`)
- **`SETUP`** — корабли видны, есть подсветка при расстановке.
- **`MY_FIELD`** — корабли видны, отображаются попадания (2) и промахи (3) противника.
- **`ENEMY_FIELD`** — корабли скрыты (рисуются как вода), видны только попадания и промахи игрока.

#### 7.1.2. Данные клеток — `int[][] cells`
- `0` — вода / не обстреляна
- `1` — корабль (виден только в режимах `SETUP` и `MY_FIELD`)
- `2` — попадание
- `3` — промах

#### 7.1.3. Отрисовка клеток (`paintCell`)
- Используются `Graphics2D`, `GradientPaint`, `RenderingHints`.
- **Вода**: вертикальный градиент от `WATER_MID` к `WATER_DEEP`, случайные точки-волны.
- **Корабль**: градиент от `SHIP_HIGHLIGHT` к `SHIP_BASE`, закруглённый прямоугольник (`fillRoundRect`), декоративная линия посередине.
- **Попадание**: красный градиент, 8-лучевая звезда (линии от центра к периферии), центр — жёлтая вспышка.
- **Промах**: вода + два концентрических круга голубого цвета и точка в центре.
- **Сетка**: тонкие линии с полупрозрачным цветом.

**Почему кастомная отрисовка вместо `JButton[][]`?**
- Производительность — 100 `JButton`'ов создают много объектов, замедляют работу.
- Анимации — с кнопками сложно делать плавные эффекты.
- Градиенты и закругления — невозможно на стандартных кнопках.
- Режимы — легко управлять видимостью кораблей одним условием.

#### 7.1.4. Подсветка при расстановке
- Метод `setHighlight()` задаёт параметры прямоугольника для предпросмотра.
- В `paintComponent()` рисуется полупрозрачная заливка зелёного (размещение допустимо) или красного (недопустимо) цвета.

#### 7.1.5. Анимации
Анимации реализованы с помощью **Swing Timer** (интервал 16 мс ≈ 60 FPS) и `AlphaComposite`.

- **Типы анимаций:** `EXPLOSION` (попадание) и `SPLASH` (промах).
- **Жизненный цикл:**
  1. При вызове `setCells()` новые клетки сравниваются со старыми; при появлении значения 2 или 3 создаётся объект `CellAnimation`.
  2. Таймер запускается, если есть анимации.
  3. Каждый тик увеличивает `progress` на 0.05 (достигает 1 за ~320 мс).
  4. В `paintAnimation()` рисуются расширяющиеся кольца и частицы с затуханием (прозрачность уменьшается).
  5. По завершении `progress >= 1` анимация удаляется, при пустом списке таймер останавливается.

**Почему Swing Timer?**
- Он выполняется в EDT — не нужен `invokeLater()`.
- Легковесный, не создаёт отдельные потоки.
- Легко управлять запуском/остановкой.

**Детерминированные искры:** для генерации случайных позиций используется `Random` с seed на основе координат клетки, что даёт одинаковые искры при каждом перерисовывании, без хранения массивов.

#### 7.1.6. Координатные метки
Номера строк и столбцов (0–9) рисуются по краям поля с помощью `drawString()` для удобства ориентирования.

### 7.2. `EventLog` — лог событий

**Файл:** `client/components/EventLog.java`

Это `JPanel`, внутри которого вертикальный список (`BoxLayout.Y_AXIS`) элементов `EventItem`. Каждый элемент — отдельная строка с цветным префиксом.

#### 7.2.1. Типы событий
- `SYSTEM` — серый, префикс «•»
- `INFO` — белый, префикс «→»
- `SUCCESS` — зелёный, префикс «✓»
- `WARNING` — золотой, префикс «⚠»
- `DANGER` — красный, префикс «✗»

Цвета и префиксы задаются в конструкторе `EventItem` через `switch`.

#### 7.2.2. Добавление события
Метод `addEvent()`:
- Создаёт `EventItem`, добавляет в `contentPanel`.
- Добавляет вертикальный отступ `Box.createVerticalStrut(4)`.
- Вызывает `revalidate()` для обновления layout.
- Прокручивает к новому элементу через `scrollRectToVisible()` (вызов в `invokeLater`, чтобы гарантировать, что элемент уже отрисован).

**Почему не `JTextArea`?** Потому что нельзя раскрасить отдельные строки. `JTextPane` с HTML — избыточно и медленнее для динамического обновления. `JPanel` + `BoxLayout` даёт полный контроль и гибкость.

### 7.3. `GradientButton` — градиентная кнопка

**Файл:** `client/components/GradientButton.java`

Наследует `JButton`, переопределяет `paintComponent()` для рендеринга с градиентом.

**Особенности:**
- Отключена стандартная заливка (`setContentAreaFilled(false)`), рамка и фокус.
- Цвета градиента передаются в конструкторе.
- При наведении (`hovered`) цвета становятся ярче (`brighter()`), при нажатии (`pressed`) — темнее (`darker()`).
- Добавлен полупрозрачный блик сверху и тонкая обводка.

**Почему кастомная?** Стандартная кнопка не поддерживает градиенты и сложные эффекты наведения. Кастомная даёт единый стиль, соответствующий дизайну.

---

### 7.4. `RoundedTextField` — скруглённое поле ввода

**Файл:** `client/components/RoundedTextField.java`

Наследует `JTextField`, переопределяет `paintComponent()` для рисования закруглённого фона и рамки.

- Фон заливается цветом `BG_PANEL` (тёмно-синий).
- Рамка меняет цвет: при фокусе — золотая (`ACCENT_GOLD`), без фокуса — серая (`BORDER_COLOR`).
- Толщина рамки при фокусе увеличена (2px вместо 1.5).

**Почему кастомный?** Стандартный `JTextField` имеет прямоугольную рамку и не позволяет легко изменить цвет фокуса и скругление. Кастомный обеспечивает единый стиль с остальным интерфейсом.

## 8. Обработка событий и потоковая модель

### 8.1. Event Dispatch Thread (EDT)

Swing — однопоточный. Все изменения GUI должны выполняться в EDT. Это гарантирует, что не будет гонок данных и мерцаний.

### 8.2. Фоновые потоки

- **Поток подключения** (`Connector`) — создаётся при нажатии кнопки в `ConnectionPanel`. Он выполняет блокирующий вызов `Socket.connect()`.
- **Поток приёма** (`receiverThread`) — запускается после успешного подключения. Он читает входящие сообщения и передаёт их в EDT через `SwingUtilities.invokeLater()`.

**Почему `invokeLater`, а не `invokeAndWait`?** Нам не нужно дожидаться завершения обработки, достаточно поставить задачу в очередь и продолжить чтение следующих сообщений.

### 8.3. Обработчики событий UI

Все слушатели (ActionListener, MouseListener, FocusListener) выполняются в EDT, т.к. они вызываются из внутреннего механизма Swing.

- **ActionListener** — для кнопок.
- **MouseListener** — для кликов по полю (расстановка, выстрел) и для палитры.
- **MouseMotionListener** — для предпросмотра при расстановке и смены курсора.
- **FocusListener** — для смены цвета рамки поля ввода.
- **InputMap/ActionMap** — для горячей клавиши R (поворот).

### 8.4. Отправка команд

Метод `send()` вызывается из EDT (обычно из ActionListener) и отправляет команду через `out.writeUTF()`. Это может блокироваться, но обычно быстро. Исключения перехватываются и игнорируются (в случае потери соединения приёмный поток обработает разрыв).

## 9. Анимации: детали реализации

Анимации в `BoardPanel` построены на **Swing Timer** и **AlphaComposite**.

**Таймер:**
```java
animTimer = new Timer(16, e -> {
    boolean needsRepaint = false;
    for (CellAnimation anim : animations) {
        anim.progress += 0.05f;
        if (anim.progress >= 1f) anim.finished = true;
        else needsRepaint = true;
    }
    animations.removeIf(a -> a.finished);
    if (needsRepaint || !animations.isEmpty()) {
        repaint();
    } else {
        animTimer.stop();
    }
});
```

- **Интервал 16 мс** обеспечивает плавность 60 кадров в секунду.
- **Шаг 0.05** даёт 20 шагов, длительность ~320 мс — оптимально для визуального эффекта.

**Прозрачность:** `alpha = 1 - progress` — от 1 до 0, создавая затухание.

**Отрисовка анимаций:** выполняется в `paintComponent()` после всех клеток, но перед сеткой и координатами. Используется `AlphaComposite.SRC_OVER` с текущей альфой. После отрисовки всех анимаций композит сбрасывается.

## 10. Выбор технологий и обоснование

| Компонент | Технология | Причина |
|-----------|------------|---------|
| GUI | Swing | Стандартная библиотека, кроссплатформенность, отсутствие внешних зависимостей. |
| Компоновка | CardLayout | Идеально для переключения между экранами, сохранение состояния. |
| Сетка | GridBagLayout | Гибкое позиционирование для форм. |
| Списки | JList + DefaultListModel + кастомный рендерер | Простота и полный контроль над отображением. |
| Графика | Java 2D (Graphics2D) | Полный контроль над рендерингом, градиенты, анимации, прозрачность. |
| Анимации | Swing Timer | Выполняется в EDT, простота управления. |
| Сеть | java.net.Socket + Data streams | Встроенная поддержка TCP, атомарное чтение строк. |
| Многопоточность | java.lang.Thread, ExecutorService | Обработка нескольких клиентов и фоновый I/O. |
| Синхронизация | synchronized, ConcurrentHashMap, AtomicInteger | Безопасная работа с общими данными. |
| Цвета | Статические константы Colors | Централизованное управление, лёгкое изменение темы. |

## 11. Заключение

Интерфейс клиента «Морской бой» спроектирован с учётом удобства пользователя и производительности. Выбор Swing и Java 2D позволил создать приятную тёмную тему с плавными анимациями без привлечения сторонних библиотек. Чёткое разделение на панели, использование `CardLayout` и потоковой модели обеспечивают надёжную и отзывчивую работу.

Все компоненты взаимосвязаны через главный класс `BattleshipClientGUI`, который управляет сетевым слоем и маршрутизацией сообщений. Благодаря этому код легко расширять (например, добавить таймеры, звуки, настройки).

Данный UI демонстрирует глубокое понимание Swing, многопоточности и Java 2D, что делает его отличным примером для изучения и дальнейшего развития.
