## Обзор проекта

Приложение состоит из двух частей:
- **Сервер** – управляет подключениями, создаёт игры, проверяет ходы и валидирует поля.
- **Клиент (GUI)** – предоставляет интерфейс для подключения, расстановки кораблей, выбора противника и ведения боя с анимациями.

Игровой процесс полностью синхронизирован через сервер, что гарантирует честность и последовательность ходов. Клиент отвечает за визуализацию и пользовательский ввод, сервер – за игровую логику.

## Технологический стек

| Компонент | Технология | Причина выбора |
|-----------|------------|----------------|
| GUI | **Swing** | Стандартная библиотека Java, не требует внешних зависимотелей, кросс-платформенная. |
| Сеть | **java.net (Socket, ServerSocket)** | Встроенный TCP/IP стек, надёжная передача данных. |
| Потоки | **java.lang.Thread, ExecutorService** | Многопоточность для одновременной обработки нескольких клиентов и фонового приёма сообщений. |
| Синхронизация | **synchronized, ConcurrentHashMap, AtomicInteger** | Потокобезопасные структуры данных и атомарные операции для конкурентного доступа. |
| Графика | **Java 2D (Graphics2D)** | Полный контроль над рендерингом: градиенты, анимации, прозрачность. |
| Анимации | **javax.swing.Timer** | Плавная анимация ~60 FPS без создания дополнительных потоков. |
| Протокол | **DataInputStream/DataOutputStream** | Атомарное чтение/запись UTF-строк, простота реализации. |

**Ключевые архитектурные паттерны:**
- **Модель-Представление (MV)** – данные (массивы `int[][]`) хранятся отдельно, `BoardPanel` только отображает их.
- **Наблюдатель** – поток-приёмник уведомляет UI через `handleMessage()`.
- **Состояние** – приложение переключается между панелями (`CONNECT`, `SETUP`, `LOBBY`, `BATTLE`), каждая имеет свой набор логических флагов (`inGame`, `myTurn`, `ready`).
- **Одиночка** – утилитарные классы `Colors` и `Constants` с приватными конструкторами.

## Сетевой протокол

Протокол **текстовый**, команды передаются в виде строк UTF-8 с разделителем `:` для параметров. Сервер и клиент используют `DataInputStream.readUTF()` / `DataOutputStream.writeUTF()` – это гарантирует атомарность чтения целых строк.

Полный список команд:

| Команда | Направление | Описание |
|---------|-------------|----------|
| `CMD_REQUIRE_FIELD` | Сервер → Клиент | Запрос поля расстановки |
| `SET_FIELD:<100×0/1>` | Клиент → Сервер | Отправка поля (строка 10×10) |
| `CMD_OK:Field accepted` | Сервер → Клиент | Поле принято |
| `CMD_ERROR:<текст>` | Сервер → Клиент | Ошибка (невалидное поле, неверный ход и т.д.) |
| `GET_PLAYERS` | Клиент → Сервер | Запрос списка доступных игроков |
| `PLAYERS:<id1>,<id2>,...` | Сервер → Клиент | Список ID готовых игроков |
| `SELECT_PLAYER:<id>` | Клиент → Сервер | Выбор противника |
| `CMD_GAME_START:<info>` | Сервер → Клиент | Игра началась |
| `CMD_YOUR_TURN` | Сервер → Клиент | Ход игрока |
| `CMD_OPPONENT_TURN` | Сервер → Клиент | Ход противника |
| `MOVE:<x>:<y>` | Клиент → Сервер | Выстрел |
| `CMD_MOVE_RESULT:<r>:<x>:<y>` | Сервер → Клиент | Результат выстрела игрока (`MISS`/`HIT`/`SUNK`/`WIN`) |
| `CMD_OPPONENT_MOVE:<r>:<x>:<y>` | Сервер → Клиент | Результат выстрела противника |
| `CMD_AGAIN` | Сервер → Клиент | Попадание, ходите ещё |
| `CMD_GAME_OVER:<WIN/LOSE>` | Сервер → Клиент | Игра завершена |
| `SURRENDER` | Клиент → Сервер | Сдача |

**Почему текстовый протокол?**
- Простота отладки (можно читать в консоли).
- Легко расширять новыми командами.
- `readUTF()`/`writeUTF()` решают проблемы с границами сообщений.

## Серверная часть

### Server.java

**Назначение:** точка входа, управление подключениями и клиентами.

**Ключевые элементы:**
- `ConcurrentHashMap<Integer, ClientHandler> clients` – потокобезопасная карта активных клиентов.
- `AtomicInteger nextId` – уникальный ID для каждого нового клиента.
- `ExecutorService pool = Executors.newCachedThreadPool()` – создаёт потоки по мере подключения и переиспользует завершённые.

**Почему `ConcurrentHashMap`?**
- Позволяет безопасно итерироваться по коллекции (`getClients()`) без блокировки всей карты.
- `synchronized` используется только на методах, изменяющих состояние (`startGame`, `removeClient`), чтобы избежать гонок при одновременных вызовах.

**Почему `newCachedThreadPool`?**
- Количество клиентов неизвестно заранее.
- Пул автоматически создаёт новые потоки при необходимости, но переиспользует завершённые, что экономит ресурсы.

**Метод `start()`:** бесконечный цикл `ServerSocket.accept()`, при каждом новом клиенте создаётся `ClientHandler`, добавляется в `clients` и отправляется в пул.

---

### ClientHandler.java

**Назначение:** один экземпляр на подключение, работает в отдельном потоке.

**Основные поля:**
- `DataInputStream in`, `DataOutputStream out` – для связи.
- `boolean[][] myField` – расстановка клиента.
- `boolean[][] opponentHits` – выстрелы по противнику.
- `Game currentGame` – текущая игра (или null).
- `boolean ready` – поле принято и валидно.

**Жизненный цикл (метод `run()`):**

1. **Получение поля (`receiveField()`)**
   - Отправляет `CMD_REQUIRE_FIELD` и ждёт `SET_FIELD:...`.
   - Парсит строку, вызывает `parseAndSetField()` → `FieldValidator.validate()`.
   - Если поле валидно – отвечает `CMD_OK`, иначе `CMD_ERROR` и повтор.
   - Устанавливает `ready = true`.

2. **Обработка команд (`processCommands()`)**
   - Бесконечный цикл чтения `in.readUTF()`.
   - Обрабатывает: `GET_PLAYERS`, `SELECT_PLAYER`, `MOVE`, `SURRENDER`.

3. **Завершение (`cleanup()`)**
   - Закрывает сокет.
   - Если игра активна, вызывает `currentGame.endGame(this, true)`.
   - Удаляет клиента из сервера.

**Метод `selectOpponent()`:**
- Проверяет, что цель существует, готова и не в игре.
- Вызывает `server.startGame(this, opp)` – синхронизированный метод, который создаёт объект `Game` и привязывает его к обоим игрокам.
- Отправляет `CMD_GAME_START` обоим и вызывает `game.notifyTurn()`.

**Метод `makeMove()`:**
- Проверяет, что игра существует, ход текущего игрока, координаты валидны, клетка не обстреляна.
- Делегирует `currentGame.makeMove(this, x, y)`.

---

### Game.java

**Назначение:** инкапсулирует логику одной игры между двумя игроками.

**Поля:** `p1, p2`, `currentTurn`, `active`.

**Метод `makeMove()`:**
1. Определяет противника `opp = (mover == p1) ? p2 : p1`.
2. Проверяет попадание: `hit = opp.myField[x][y]`.
3. Записывает выстрел в `mover.opponentHits[x][y] = true`.
4. Определяет результат:
   - `MISS` – если не попал.
   - `HIT` – попал, но корабль не потоплен.
   - `SUNK` – попал и все клетки корабля подбиты (проверка через `isShipSunk()`).
5. Проверяет победу: `checkWin()` – все ли корабли противника подбиты.
6. Отправляет результаты обоим игрокам.
7. Если победа – завершает игру.
8. Если попадание – отправляет `CMD_AGAIN` (дополнительный ход), иначе меняет `currentTurn` и уведомляет о ходе.

**Поиск корабля (`isShipSunk`):**
- Использует **рекурсивный DFS** (в 4 направлениях) для поиска всех клеток корабля по точке попадания.
- Проверяет, все ли клетки помечены как подбитые в `hits`.
- DFS гарантирует, что будут найдены только клетки одного корабля (благодаря валидатору, корабли не касаются друг друга).

**Почему рекурсия?**
- Код компактнее и нагляднее, глубина рекурсии ≤ 4 (длина корабля), стек не переполнится.

---

### FieldValidator.java

**Назначение:** статический утилитарный класс для проверки корректности расстановки.

**Алгоритм `validate()`:**
1. Обходит всё поле.
2. При нахождении клетки корабля, не помеченной как `visited`, запускает DFS (`findShipCells`) для поиска всех клеток этого корабля.
3. Проверяет:
   - размер корабля от 1 до 4;
   - прямолинейность (`isShipStraight`) – все клетки имеют одинаковую строку или столбец;
   - отсутствие касаний с другими кораблями (`hasNoTouching`) – проверяет 8 соседей для каждой клетки.
4. Увеличивает счётчик `counts[size]`.
5. После обхода сравнивает `counts` с эталоном (1×4, 2×3, 3×2, 4×1).

**Важно:** проверка касаний осуществляется на уровне сервера, чтобы клиент не мог отправить невалидное поле (даже если в его UI есть предпроверка).

## Клиентская часть

### BattleshipClientGUI.java

**Назначение:** главное окно, управляет жизненным циклом клиента.

**Ключевые компоненты:**
- `CardLayout` + `JPanel mainPanel` – переключение между 4 панелями.
- `Socket`, `DataInputStream`, `DataOutputStream` – сетевое соединение.
- `Thread receiverThread` (daemon) – фоновый поток для чтения сообщений.
- `volatile boolean connected` – флаг состояния.

**Метод `connect()`:**
- Запускает поток `Connector`, создающий сокет.
- При успехе инициализирует `in`/`out`, устанавливает `connected=true`, запускает `receiverThread`.
- При ошибке – возвращает управление в `ConnectionPanel`.

**Метод `receiveLoop()`:**
- Пока `connected` – читает `in.readUTF()` и через `SwingUtilities.invokeLater()` передаёт сообщение в `handleMessage()`.
- При `IOException` – разрывает соединение и переключает на панель подключения.

**Метод `handleMessage()` – диспетчер:**
- Разбирает команды и вызывает соответствующие методы панелей.
- Все обновления UI выполняются в потоке EDT (благодаря `invokeLater`).

**Почему `invokeLater`?**
- Swing – не потокобезопасен. Все изменения GUI должны выполняться в Event Dispatch Thread.
- `receiverThread` – фоновый, поэтому необходимо явно передавать управление EDT.

**Почему daemon-поток?**
- Если основной поток завершится, JVM не будет ждать завершения daemon-потоков, что ускоряет выход.

## Взаимодействие компонентов клиента

1. **Поток приёма сообщений (`receiverThread`)** непрерывно читает команды из `in.readUTF()` и передаёт их в `handleMessage()` через `SwingUtilities.invokeLater()`.
2. **`handleMessage()`** – центральный диспетчер:
   - Команды от сервера сопоставляются с методами соответствующих панелей.
   - Например, `CMD_MOVE_RESULT` вызывает `battlePanel.applyMyMove()`, которая обновляет `enemyCells` и `BoardPanel`.
3. **Потокобезопасность:** все вызовы методов панелей происходят в EDT, т.к. `handleMessage()` выполняется после `invokeLater()`.
4. **Обратная связь:** пользовательский ввод (клики, нажатия кнопок) также обрабатывается в EDT и вызывает `client.send()` – отправку команды в фоновом потоке (но без блокировки UI).
5. **Синхронизация состояния:** панели используют флаги (`inGame`, `myTurn`, `ready`) для контроля доступных действий.

## Детали реализации

### Потокобезопасность

- **Сервер:**
  - `ConcurrentHashMap` для клиентов – безопасная итерация.
  - `synchronized` на `startGame()` и `removeClient()` – защита от гонок.
  - `AtomicInteger` для ID – атомарный инкремент.
- **Клиент:**
  - Флаг `connected` объявлен как `volatile`, чтобы гарантировать видимость изменений между потоками.
  - Все обновления GUI – через `SwingUtilities.invokeLater()`.

### Обработка сообщений

Сообщения от сервера обрабатываются последовательно в потоке EDT, что гарантирует отсутствие конкурентных изменений UI. При этом отправка сообщений (`client.send()`) происходит в том же потоке, что и вызов (обычно это EDT), но сам `writeUTF()` не блокирует надолго.

### Анимации в BoardPanel

- Используется `javax.swing.Timer` с задержкой 16 мс (≈60 FPS).
- При обновлении клеток (попадание/промах) создаётся объект `CellAnimation` с типом и начальным `progress = 0`.
- В каждом тике таймера `progress` увеличивается на 0.05 (плавность за 20 шагов).
- Когда `progress >= 1`, анимация удаляется.
- Если анимаций нет, таймер останавливается автоматически для экономии ресурсов.
- В `paintComponent()` анимации рисуются поверх клеток с использованием `AlphaComposite` для затухания.

### Валидация поля (сервер)

- Клиент может отправить любое поле, даже если в UI он не давал его расставить (например, модифицированный запрос).
- Серверная валидация обязательна: проверяются все правила (касания, количество кораблей, прямолинейность).
- Если поле невалидно – клиент получает `CMD_ERROR` и должен расставить заново.

### Автоматическая расстановка (клиент)

- Алгоритм: перемешивает список размеров, для каждого размера делает до 5000 попыток случайного размещения с проверкой `canPlace()`.
- Если не удалось разместить корабль за 5000 попыток, рекурсивно перезапускает генерацию (это гарантирует, что поле будет валидно, т.к. пространство решений конечно).

## Сценарий игры

Ниже приведена полная последовательность действий игрока и соответствующие сетевые взаимодействия.

1. **Подключение** (ConnectionPanel)
   - Пользователь вводит хост и порт, нажимает «Подключиться».
   - Клиент создаёт сокет, устанавливает соединение.

2. **Получение запроса поля** (сервер → клиент)
   - Сервер отправляет `CMD_REQUIRE_FIELD`.
   - Клиент переключается на `SetupPanel`.

3. **Расстановка кораблей** (SetupPanel)
   - Пользователь расставляет корабли (вручную или авто).
   - Нажимает «Готов» → клиент отправляет `SET_FIELD:0101...`.
   - Сервер валидирует поле → если OK, отвечает `CMD_OK`.

4. **Лобби** (LobbyPanel)
   - Клиент переключается на панель лобби и автоматически отправляет `GET_PLAYERS`.
   - Сервер возвращает `PLAYERS:2,5,7`.
   - Пользователь выбирает противника и нажимает «Вызвать на бой» → `SELECT_PLAYER:2`.

5. **Начало игры** (сервер → оба клиента)
   - Сервер создаёт `Game`, отправляет `CMD_GAME_START:...` обоим игрокам.
   - Оба клиента переключаются на `BattlePanel`.

6. **Ход игрока A** (BattlePanel)
   - Игрок A получает `CMD_YOUR_TURN`, его поле противника становится кликабельным.
   - Он кликает по клетке → отправляет `MOVE:3:5`.
   - Сервер обрабатывает ход:
     - `MISS` – отправляет `CMD_MOVE_RESULT:MISS:3:5` A и `CMD_OPPONENT_MOVE:MISS:3:5` B.
     - `HIT` – отправляет аналогично, но с `HIT` и затем `CMD_AGAIN` A.
     - `SUNK` – отправляет `SUNK`.
     - `WIN` – отправляет `WIN` и после этого `CMD_GAME_OVER`.

7. **Ответные действия клиентов:**
   - A обновляет `enemyCells`, B обновляет `myCells`.
   - Оба получают анимации на своих `BoardPanel`.

8. **Победа/поражение**
   - Сервер отправляет `CMD_GAME_OVER:WIN` победителю и `CMD_GAME_OVER:LOSE` проигравшему.
   - Оба клиента возвращаются в лобби.

## Заключение

Проект представляет собой полноценное клиент-серверное приложение для игры в морской бой. В нём учтены:

- **Масштабируемость сервера:** пул потоков, потокобезопасные структуры данных.
- **Надёжность протокола:** текстовые команды с чёткой семантикой.
- **Качественный UI:** тёмная тема, плавные анимации, интуитивные подсказки.
- **Защита от ошибок:** серверная валидация, проверка ходов, блокировка некорректных действий.

Архитектурные решения (Swing, Java 2D, многопоточность, синхронизация) выбраны с учётом простоты, производительности и переносимости. Код полностью готов к расширению (например, добавление таймеров, звуков, сохранения статистики).
