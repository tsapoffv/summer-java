3. Серверная часть
3.1 Server.java — главный класс сервера
Файл: src/server/Server.java
Назначение: точка входа сервера, приём TCP-подключений, управление пулом клиентов, создание игр.
Поля:
- ConcurrentHashMap<Integer, ClientHandler> clients — потокобезопасная карта всех активных клиентов
- AtomicInteger nextId — уникальный атомарный счётчик для ID клиентов
- static final int PORT = 12345 — порт сервера
- static final int BOARD_SIZE = 10 — размер поля
- static final int[] SHIP_SIZES = {4,3,3,2,2,2,1,1,1,1} — эталонные размеры кораблей
Методы:
- main() → new Server().start()
- start() — создаёт ServerSocket, бесконечный цикл accept(), пул newCachedThreadPool()
- synchronized removeClient(int id) — удаляет клиента из карты
- synchronized startGame(ClientHandler a, ClientHandler b) — проверяет готовность, создаёт Game
Потоковая модель:
main thread:
  └─ ServerSocket.accept() (блокирующий)
       └─ new ClientHandler → pool.execute(handler)
Почему newCachedThreadPool(): Количество клиентов динамическое. Пул создаёт потоки по мере подключения и переиспользует завершённые. В отличие от fixedThreadPool, не требует заранее знать максимальное число клиентов.
Почему synchronized на startGame(): Два клиента могут одновременно вызвать selectOpponent() с одинаковой целью. Без синхронизации возможна ситуация, когда один и тот же игрок оказывается в двух играх сразу.
Почему ConcurrentHashMap: Позволяет безопасно итерироваться (getClients() для sendPlayerList()) без блокировки всей карты. Конкурентные чтения/вставки не блокируют друг друга (в отличие от Collections.synchronizedMap).

3.2 ClientHandler.java — обработчик клиента
Файл: src/server/ClientHandler.java
Назначение: один экземпляр = один подключённый клиент. Работает в отдельном потоке (implements Runnable).
Поля:
- Socket socket — TCP-соединение
- DataInputStream in / DataOutputStream out — ввод/вывод
- int id — уникальный ID
- Server server — ссылка на сервер
- boolean[][] myField — поле клиента (true = корабль)
- boolean[][] opponentHits — карта выстрелов по противнику
- Game currentGame — текущая игра (null если не в игре)
- boolean ready — готовность (поле принято)
Жизненный цикл (метод run()):
run()
  ├── receiveField()           # Этап 1: Получение поля
  │     └── Цикл:
  │           out.writeUTF("CMD_REQUIRE_FIELD")
  │           in.readUTF() → "SET_FIELD:..."
  │           parseAndSetField(fieldData)
  │           FieldValidator.validate(field) → OK?
  │           out.writeUTF("CMD_OK" / "CMD_ERROR")
  │
  ├── processCommands()        # Этап 2: Обработка команд
  │     └── Бесконечный цикл:
  │           in.readUTF()
  │           ├── "GET_PLAYERS"     → sendPlayerList()
  │           ├── "SELECT_PLAYER:"  → selectOpponent()
  │           ├── "MOVE:x:y"        → makeMove()
  │           └── "SURRENDER"       → surrender()
  │
  └── cleanup()                # finally: закрытие сокета,
                                # endGame(), server.removeClient()
Ключевые методы:
receiveField(): Сервер активно запрашивает поле. Клиент не может начать игру, пока не отправит валидное поле. Если поле невалидно → сервер запрашивает снова.
selectOpponent(targetId):
1. Проверка: currentGame == null, цель существует, opp.ready, opp.currentGame == null
2. server.startGame(this, opp) — синхронизированный вызов
3. При успехе → обоим CMD_GAME_START, затем currentGame.notifyTurn()
makeMove(x, y):
1. Проверка: currentGame != null, currentGame.currentTurn == id, координаты в 0..9, !opponentHits[x][y]
2. currentGame.makeMove(this, x, y) — делегирование логики Game
parseAndSetField(data):
- Строка 100 символов → boolean[10][10]
- Проверка: data.length() == 100
- FieldValidator.validate(field) — кастомная валидация
- Создание opponentHits = new boolean[10][10]

3.3 Game.java — логика игры
Файл: src/server/Game.java
Назначение: одноразовый объект для одной игры между двумя игроками.
Поля:
- ClientHandler p1, p2 — игроки
- int currentTurn — ID текущего игрока
- boolean active — флаг активности (защита от двойного завершения)
Методы:
makeMove(mover, x, y):
1. opp = (mover == p1) ? p2 : p1
2. hit = opp.getMyField()[x][y]
3. mover.getOpponentHits()[x][y] = true
4. if (!hit) → result = "MISS"
   else     → result = isShipSunk() ? "SUNK" : "HIT"
5. win = checkWin(opp, mover.getOpponentHits())
6. mover  ← "CMD_MOVE_RESULT:result:x:y"
   opp    ← "CMD_OPPONENT_MOVE:result:x:y"
7. if (win) → endGame(opp, false)
8. if (hit) → mover ← "CMD_AGAIN"
   else     → currentTurn = opp.getId(); notifyTurn()
isShipSunk(field, hits, x, y):
1. visited = boolean[10][10]
2. DFS findShipCells(field, visited, x, y, shipCells)
3. for (cell : shipCells):
     if (!hits[cell]) return false
   return true
findShipCells(): Рекурсивный DFS в 4 направлениях. Находит все связанные клетки корабля, начиная с точки попадания.
checkWin(opp, hits): Проходит по всему полю противника. Если есть живая клетка корабля (true в поле, false в hits) → не победа.
endGame(loser, surrender):
if (!active) return        # защита от повторного вызова
active = false
winner ← "CMD_GAME_OVER:WIN"
loser  ← "CMD_GAME_OVER:LOSE"
p1.setCurrentGame(null)
p2.setCurrentGame(null)
Почему DFS для findShipCells: Корабль может быть расположен горизонтально или вертикально. DFS в 4 стороны (без диагоналей) гарантированно обходит все клетки одного корабля, игнорируя соседние (валидатор проверяет отсутствие касаний, так что DFS не перейдёт на другой корабль).

3.4 FieldValidator.java — валидация поля
Файл: src/server/FieldValidator.java
Назначение: статический утилитарный класс для валидации расстановки кораблей.
Логика validate(field):
1. Проход по всем клеткам поля
2. Если клетка == true и не visited:
     findShipCells(field, visited, i, j, shipCells)
     size = shipCells.size()
     if (size == 0 || size > 4) → false
     counts[size]++
     if (!isShipStraight(shipCells)) → false
     if (!hasNoTouching(field, shipCells)) → false
3. Сверка counts с expected {0,1,2,3,1} (1×4, 2×3, 3×2, 4×1)
4. return true/false
isShipStraight(cells): Все клетки имеют одинаковую X (горизонтальный) или одинаковую Y (вертикальный). Если хоть одна отличается по обеим осям → не прямолинейно.
hasNoTouching(field, shipCells): Для каждой клетки корабля проверяет 8 соседей. Если сосед = true и не входит в этот же корабль → касание, поле невалидно.
Почему отдельный класс: Валидация не зависит от состояния сервера (чистая функция). Статический метод можно вызывать из любого контекста. Легко тестировать.

Файл: src/client/BattleshipClientGUI.java
Назначение: JFrame, точка входа, управление экранами, сетевое соединение, маршрутизация сообщений.
Ключевые компоненты:
Компонент	Тип	Назначение

mainPanel	JPanel	Контейнер с CardLayout
socket	Socket	TCP-соединение с сервером
in / out	DataInputStream / DataOutputStream	Сетевой ввод/вывод
receiverThread	Thread (daemon)	Фоновый поток для чтения сообщений
Сетевая модель:
connect(host, port):
  new Thread("Connector") {
    socket = new Socket(host, port)
    in = new DataInputStream(socket.getInputStream())
    out = new DataOutputStream(socket.getOutputStream())
    connected = true
    receiverThread = new Thread(this::receiveLoop)
    receiverThread.setDaemon(true)
    receiverThread.start()
  }
receiveLoop():
  while (connected) {
    msg = in.readUTF()
    SwingUtilities.invokeLater(() → handleMessage(msg))
  }
handleMessage() — диспетчер сообщений:
CMD_REQUIRE_FIELD   → setupPanel.reset(); showPanel("SETUP")
CMD_OK              → showPanel("LOBBY"); lobbyPanel.refreshPlayers()
CMD_ERROR:*         → игнорируется (обрабатывается на панелях)
PLAYERS:*           → lobbyPanel.updatePlayers(list)
CMD_GAME_START:*    → battlePanel.initGame(info); showPanel("BATTLE")
CMD_YOUR_TURN       → battlePanel.setMyTurn(true)
CMD_OPPONENT_TURN   → battlePanel.setMyTurn(false)
CMD_MOVE_RESULT:*   → battlePanel.applyMyMove(x, y, result)
CMD_OPPONENT_MOVE:* → battlePanel.applyOpponentMove(x, y, result)
CMD_AGAIN           → battlePanel.showAgain()
CMD_GAME_OVER:*     → battlePanel.endGame(); showPanel("LOBBY"); refreshPlayers()
Почему SwingUtilities.invokeLater(): Весь сетевой ввод происходит в потоке receiverThread. Swing не thread-safe — все манипуляции с GUI должны выполняться в Event Dispatch Thread (EDT). invokeLater() ставит задачу в очередь EDT.
Почему CardLayout: Позволяет хранить 4 разных экрана в одном контейнере и переключаться между ними без создания новых окон.
main():
1. Настройки Linux (AWT toolkit, AA-текст, uiScale)
2. UIManager.setLookAndFeel(system L&F)
3. check headless → exit
4. SwingUtilities.invokeLater(() → new BattleshipClientGUI().setVisible(true))

4.6 BoardPanel.java — кастомная отрисовка поля
Файл: src/client/components/BoardPanel.java
Самая технически сложная часть клиента. Полностью переопределяет paintComponent().
Режимы (Mode):
- SETUP — корабли видны + подсветка расстановки
- MY_FIELD — корабли видны + результаты выстрелов противника
- ENEMY_FIELD — корабли скрыты (как вода), видны только свои выстрелы
Рендеринг клеток (paintCell):
state	Визуал	Java 2D техника
0 (вода)	Синий градиент	GradientPaint(WATER_MID → WATER_DEEP), fillRect
1 (корабль)	Серый градиент, скруглённые углы	GradientPaint(SHIP_HIGHLIGHT → SHIP_BASE), fillRoundRect
2 (попадание)	Красный градиент + взрывная звезда	8-лучевая звезда через drawLine, центр fillOval
3 (промах)	Вода + синее кольцо + точка	drawOval, fillOval
Система анимации (Swing Timer, 16ms ≈ 60 FPS):
CellAnimation { row, col, type(EXPLOSION|SPLASH), progress 0.0→1.0, finished }
setCells(newCells):
  └─ сравнивает old vs new
       ├─ 0→2: add EXPLOSION
       ├─ 0→3: add SPLASH
       └─ start Timer
Timer tick (16ms):
  anim.progress += 0.05
  removeIf(finished)
  repaint()
  if empty → stop
paintAnimation():
  AlphaComposite.SRC_OVER (alpha = 1 - progress)
  EXPLOSION: расширяющиеся кольца + искры (Random с seed)
  SPLASH: расширяющиеся водяные кольца
Подсветка расстановки: Зелёный (можно) / красный (нельзя), полупрозрачная заливка.
Координатные метки: drawString по краям поля.
Почему кастомная отрисовка вместо JButton[][]:
1. Производительность: 100 клеток × JButton = 100 компонентов. Кастомная — один paintComponent().
2. Анимации: Стандартные компоненты не поддерживают анимации.
3. Градиенты: GradientPaint невозможен с JButton.
4. Скрытие: В режиме ENEMY_FIELD корабли не рисуются — одно условие в paintCell().
